## 2. 문제 현황

### 재현 파일
`samples/exam_eng.hwp` 8 페이지 중 5 페이지, 우측 단(단 1) 의 문단 0.221 (`32.- Education, ...`).

해당 문단 line_seg[10] (ts=600~664) — 빈칸 dash 시퀀스 + " of being 'stimulus-driven',".

### 측정값 (현재 SVG, devel)

| 항목 | 측정값 | 근거 |
|------|-------|------|
| dash 1개 advance | **12.11 px** | `<text x=597.12...>` `<text x=609.23...>` 간격 |
| dash 시퀀스 시각 폭 | **~351 px** (29 dash × 12.11) | 597.12 → 936.18 + 12.11 |
| "of" 시작 x | **953.04** | 단 1 우측 끝(약 1015 px) 가까움 |
| 빈칸 underline | x1=597.12, x2=948.29 | `<line>` 태그 |

### PDF 기대값 (이슈 인용)

| 항목 | 기대값 |
|------|-------|
| dash 1개 advance | ~3.46 px (HY신명조 384/1024 em × 0.95 ratio + spacing −13%) |
| 빈칸 시각 폭 | ~135 px |
| "of being" x | 단 중앙쯤 |

### 증상

- "of being 'stimulus-driven'," 의 우측 끝(`,`)이 단 우측 경계를 넘어 잘림
- 빈칸 dash 시각상 글리프 사이 갭이 노출 (HY신명조 dash 글리프 폭 5.46 px ≪ advance 12.11 px)

## 3. 원인 분석

### 직접 원인 (가설, 단계 1 에서 확정 예정)

`src/renderer/layout/text_measurement.rs:209` 의 advance 공식
```rust
let mut w = base_w * ratio + style.letter_spacing + style.extra_char_spacing;
```
에서 dash 의 최종 advance 가 12.11 px. 역산:
- 자연 advance 기대 = base_w(5.74) × 0.95 + (−1.99) = 3.47 px
- 측정 12.11 px → 추가 spread 8.64 px/char

후보 1: **Justify Branch B 발동** (`paragraph_layout.rs:1024-1031`)
- 분기 조건: `interior_spaces == 0` (라인 내 ASCII space 가 없을 때)
- 발동 시 모든 글자에 `extra_char_sp = (avail − text_w) / total_char_count` 균등 분배
- 8.64 px ≈ Branch B 분배값과 일치 → 가장 유력

후보 2: **메트릭 부재 시 폴백** — `measure_char_width_embedded` 가 HY신명조 dash 에 대해 None → `font_size * 0.5 = 7.65 px` (still not 12.11 px, 단독으로는 부족)

후보 3: **메트릭 자체 큰 값** — HY신명조 메트릭이 dash 에 대해 ~em 수준 폭 반환 (단독 가능 시나리오)

### 왜 Branch B 가 발동했는가 (가설)

dash 시퀀스 + "of being..." 이 같은 HWP LINE_SEG[10] 에 묶여 있으나, **렌더러 내부 ComposedLine 구성 시 두 개로 분할**되었을 가능성. 분할 사유:
- 폰트 분기: dash 는 HY신명조(CJK 카테고리) 로, 영문 "of being" 은 Times New Roman(Latin) 으로 구분되어 run/line 단위가 잘림
- 또는 음수 indent 와 음수 spacing 누적으로 dash run 만 너비 초과 판정되어 단독 라인 처리

dash-only ComposedLine 의 경우 ASCII space 가 없어 Branch B 발동 → spread 8.64 px/char.

### 가운데점(·) 처리 패턴과의 비교

`svg.rs:1882-1932` 의 `is_middle_dot` → `<circle>` 대체 패턴은 **시각만** 변경하고 advance 는 정상값 유지(가운데점 메트릭 정상). dash 사례는 advance 자체가 부풀려져 있으므로 시각 대체만으로는 "of being" 잘림이 해소되지 않음.

## 4. 수정 방향 (Option C)

### 4-A. advance 자연 폭 보정 (레이아웃 fix)

dash run (또는 leader-like 반복 글자) 가 Justify spread 에 포함되지 않도록 한다.

설계 후보 (구현계획서에서 1개 선택):
- **(a)** Justify 분배에서 leader-like 반복 글자 카운트 제외: `total_char_count` 산정 시 `[c1, c2, c3, ...]` 같은 dash 반복 시퀀스를 `1` 로 압축 카운트하고, `total_text_width` 도 자연 폭으로 누적
- **(b)** Branch B 자체에서 dash 류를 균등 분배 대상에서 제외 (`if c == '-' { skip_spread }` 패턴)
- **(c)** 분기 결정에서 leader-like 시퀀스를 단어 경계처럼 취급해 Branch A 가 발동하도록 가상 공백 카운트 추가

기준: 메트릭 정상(자연 advance 3.47 px) 이라는 단계 1 결과를 가정. 다르면 4-A 설계 변경.

### 4-B. dash run 시각 통합 (SVG/Canvas 렌더 fix)

연속 dash 시퀀스를 단일 `<line>`/`<rect>` 로 그려 글리프 사이 갭을 제거한다. 가운데점 패턴(`svg.rs:1924`) 모방.

- 클러스터 순회 시 연속 dash 감지 → run 시작 x, 끝 x 계산
- `<line x1=start y1=baseline_offset x2=end y2=baseline_offset stroke-width=...>` 출력
- 해당 dash 글리프들은 `<text>` 출력 스킵 (개별 글리프 미렌더)
- `web_canvas.rs` 도 동일 패턴 적용

advance 자체는 4-A 결과(자연 폭) 를 사용하므로 후속 글자 위치도 자동 정상화.
