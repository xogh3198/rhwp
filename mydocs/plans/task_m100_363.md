# Task M100 #363 수행계획서: 기존 PageRenderTree 구현의 native bridge API 노출

## 1. 작업 개요

- **Issue**: [#363 기존 PageRenderTree 구현의 native bridge API 노출 제안](https://github.com/edwardkim/rhwp/issues/363)
- **브랜치**: `feature/issue-363-native-render-tree-api`
- **기준**: `upstream/devel` (`75496e9` 기준)
- **작업 방식**: 외부기여자 workflow (`origin` fork feature 브랜치 → upstream `devel` 대상 PR)

## 2. 목표

native bridge에서 WASM 경로를 우회하지 않고 Rust core의 기존 page render tree를 직접 사용할 수 있도록 public API 경계를 정리한다.

필수 API는 다음 두 가지다.

```rust
pub fn build_page_render_tree(
    &self,
    page_num: u32,
) -> Result<PageRenderTree, HwpError>

pub fn get_bin_data(
    &self,
    index: usize,
) -> Option<&[u8]>
```

## 3. 배경 및 전제

- `PageRenderTree` 생성 경로는 이미 내부 구현과 WASM API에서 사용되고 있다.
- native viewer는 SVG 문자열을 다시 rasterize하지 않고 `PageRenderTree`를 platform model로 변환해 CoreGraphics/CoreText 등으로 렌더링할 수 있어야 한다.
- 이미지 노드는 `bin_data_id`만 참조하므로, native renderer가 실제 이미지를 디코드하려면 별도 binary lookup API가 필요하다.
- 이슈 본문에 언급된 검증 커밋 `1e9d78a Add native render tree bridge APIs`는 로컬 `devel`에 존재하지만, 이번 작업은 최신 `upstream/devel` 기준으로 필요한 변경을 재정리한다.

## 4. 수행 범위

1. `DocumentCore` public API로 `build_page_render_tree(page_num)`를 추가한다.
2. `DocumentCore` public API로 `get_bin_data(index)`를 추가한다.
3. native bridge에서 `PageRenderTree`를 JSON 직렬화할 수 있도록 필요한 최소 `serde::Serialize` derive를 정리한다.
4. `ImageNode.bin_data_id`와 `get_bin_data(index)`의 index 기준을 문서화한다.
5. 기존 SVG / HTML / Canvas / WASM 렌더링 경로의 동작 변경이 없는지 검증한다.

## 5. 제외 범위

- 새로운 render tree 구조 설계
- 새로운 native renderer 구현
- SVG / HTML / Canvas 렌더러의 출력 포맷 변경
- 이미지 binary를 `PageRenderTree` JSON에 직접 포함하는 방식
- HWPX serializer 또는 편집 API 확장

## 6. 조사 계획

1. 최신 `upstream/devel`에서 `DocumentCore`, render tree, WASM `getPageRenderTree` 경로를 확인한다.
2. `bin_data_id`가 생성되는 위치와 문서 내부 `bin_data_content`의 인덱스 기준을 확인한다.
3. 기존 검증 커밋 `1e9d78a`의 변경을 참고하되, 현재 upstream 구조에 맞게 필요한 부분만 재적용한다.
4. 직렬화 derive 추가 범위를 최소화해 public API 표면과 기존 renderer 영향 범위를 제한한다.

## 7. 검증 계획

- `cargo build`
- `cargo test`
- 필요 시 `cargo test render_tree` 등 관련 단위 테스트 범위 확인
- 문서화 변경은 링크와 명칭 규약을 점검한다.

## 8. 리스크 및 대응

| 리스크 | 대응 |
|--------|------|
| `PageRenderTree` 내부 타입 중 일부가 `Serialize`를 지원하지 않아 derive 전파가 넓어질 수 있음 | native bridge에 필요한 타입만 확인하고 최소 범위로 derive 추가 |
| `build_page_render_tree`가 내부 overflow 상태를 변경할 가능성 | 기존 WASM/렌더링 경로와 동일한 정리 동작인지 확인하고 테스트로 회귀 감시 |
| `bin_data_id`와 `bin_data_content` index 기준 불일치 가능성 | 생성부와 lookup부를 함께 추적하고 문서에 기준 명시 |
| 최신 upstream에서 검증 커밋과 구조가 달라졌을 가능성 | `1e9d78a`를 그대로 cherry-pick하지 않고 현재 코드 기준으로 수동 반영 |

## 9. 승인 요청
