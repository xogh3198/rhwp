## 2. 원인 (선행 분석 결론)

`src/renderer/typeset.rs:421-451` 의 #359 선제 가드가 다단의 **단 전환** 도 "단독 항목 페이지 위험" 으로 오인하여 발동.

- #321 vpos-reset 가드 (line 409) 는 `advance_column_or_new_page()` 호출 → `col_count` 검사로 다단의 비-마지막 단에서는 다음 단으로만 이동 (페이지 추가 없음).
- #359 선제 가드는 `current_column + 1 < col_count` 검사 부재 → 다단의 모든 단 경계에서 오발동.

추가로 누적 공식 (`current_height += total_height`) 이 다단 단 채움을 ~수십 px 인플레이션 (보조 요인).

다단 판정자는 `st.col_count > 1` 로 통일 (typeset.rs:533, 746 / pagination engine.rs:607 동일).

## 4. 변경 범위

**최소 침습 — 단일 함수 한 곳**:

`src/renderer/typeset.rs` 의 `typeset_section` 내 `next_will_vpos_reset` 계산 (line 431~444) 에 가드 비활성화 조건 추가:

```rust
let is_last_column = st.current_column + 1 >= st.col_count;
let next_will_vpos_reset = if !st.current_items.is_empty()
    && para_idx + 1 < paragraphs.len()
    && is_last_column                       // ← 추가
{
    // 기존 로직
};
```

비-마지막 단에서는 #321 가드 (line 409) 가 다음 iteration 에 정상 발동하여 `advance_column_or_new_page()` 로 단 전환을 처리.

누적 공식 (`current_height += total_height`) 은 **건드리지 않음**. 가드 수정만으로 단 전환이 정상화되면 단 채움 비대칭이 자연스럽게 해소될 것으로 예상. 부족 시 단계 4 에서 추가 검토.

## 6. 리스크

- **단계 2 만으로 부족할 가능성**: 누적 공식 변경분 ~91px 가 잔존하여 col 0 채움이 여전히 hwp_used 대비 부족할 수 있음. 이 경우 단계 3 에서 단 채움 진단 후 단계 4 에서 누적 공식의 다단 분기 (`is_last_column ? total_height : height_for_fit`) 를 추가 적용 검토.
- **k-water-rfp 회귀 위험**: 가드 자체가 k-water-rfp 의 단단 (single-column) 케이스에 대해 작동하던 부분 — `is_last_column` 조건은 단단 (`col_count == 1`) 에서는 항상 true 이므로 단단 거동 변화 없음. 회귀 위험 낮음.
- **hwp-multi-001 (다단 + 페이지 분할 테스트)**: 다단 마지막 단에서는 가드가 그대로 발동하므로 영향 없음. force_page_break 분기도 변경 없음.
