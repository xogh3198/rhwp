## 3. 근본 원인

`layout.rs:1392-1430` 의 인접 문단 vpos 보정 공식:

```rust
let vpos_end = seg.vertical_pos + seg.line_height + seg.line_spacing;
let end_y = col_area.y + hwpunit_to_px(vpos_end - base, self.dpi);
if end_y >= col_area.y && end_y <= col_area.y + col_area.height
    && end_y >= y_offset - 1.0
{ y_offset = end_y; }
```

`base = 첫 항목의 first seg vpos` 인데, sequential 배치에서 첫 항목은 `col_area.y + first_vpos * scale` 위치에 놓임 (절대 vpos 좌표 사용). 따라서:

- 첫 항목의 실제 y = `col_area.y + base * scale`
- 보정 공식의 end_y = `col_area.y + (vpos_end - base) * scale` = (실제 vpos 위치) − `base * scale`
- → 보정값이 `base * scale` 만큼 위로 어긋남

좌측 단은 base=2144 (≈ 28.6 px) 라서 어긋남이 작아 조건 `end_y >= y_offset - 1.0`을 우연히 통과. 우측 단은 base=7060 (≈ 94.2 px) 이라 어긋남이 커서 보정이 sequential y_offset보다 작아져 스킵.

## 4. 수정 방향 (옵션 비교)

### 옵션 A — 공식에서 base 의미를 일치시키기 (권장)

`col_area.y` 가 vpos=0 을 의미하는 것이 자연스러우므로 (sequential 배치가 그렇게 동작), **base 차감을 제거**:

```rust
let end_y = col_area.y + hwpunit_to_px(vpos_end, self.dpi);
```

또는 `base` 보존 시 첫 항목 보정을 함께 더함:

```rust
let end_y = col_area.y + hwpunit_to_px(vpos_end, self.dpi);
// = col_area.y + (vpos_end - base) * scale + base * scale
```

영향:
- 좌측 단: 기존 보정값보다 `base * scale` (≈28.6 px) 만큼 커짐. 그러나 sequential y_offset 도 이미 그만큼 누적된 절대 vpos 기준이라 조건 통과 가능 (재검증 필요).
- 우측 단: 보정값이 sequential 범위 안으로 들어와 조건 통과 → 정상 작동.

### 옵션 B — vpos_page_base 정의 변경

`vpos_page_base = 0` (column-top vpos) 으로 둬서 base 차감을 무의미화. 코드 의미 일관성은 유지되나, lazy_base 로직(`vpos_lazy_base`)과의 역할 분리가 흐려짐.

### 채택안

**옵션 A**: 변경 범위가 최소이고 sequential 배치(이미 정확) 와 보정 공식의 일치를 직접 이룸. `vpos_page_base` / `vpos_lazy_base` 의 존재 의미가 사라지면 함께 정리.

다만 lazy_base 분기는 sequential 자체를 신뢰할 수 없을 때(자리차지 표 등) 역산 보정용이므로 분리 유지 가능. → 구현 단계에서 결정.
