# Task #508 수행 계획서 — PageLayerTree image paint op brightness/contrast 직렬화

## 이슈 정합

- **Issue**: [#508](https://github.com/edwardkim/rhwp/issues/508) — PageLayerTree image paint op에 brightness/contrast 필드 추가 필요
- **작성자**: @postmelee
- **마일스톤**: GitHub issue 미지정. 오늘할일 기준 `M100 — v1.0.0 조판 엔진`으로 분류
- **대표 샘플**: `samples/복학원서.hwp`
- **대표 증상**: `PageRenderTree` 기반 renderer는 `ImageNode.brightness/contrast`를 읽을 수 있지만, `PageLayerTree` JSON 기반 downstream renderer는 같은 이미지 보정 값을 알 수 없음

## 현재 코드 정합

`ImageNode`에는 필요한 필드가 이미 존재한다.

| 필드 | 현재 위치 | 기본값 |
|------|-----------|--------|
| `effect` | `src/renderer/render_tree.rs::ImageNode` | `ImageEffect::RealPic` |
| `brightness` | `src/renderer/render_tree.rs::ImageNode` | `0` |
| `contrast` | `src/renderer/render_tree.rs::ImageNode` | `0` |
| `crop` | `src/renderer/render_tree.rs::ImageNode` | `None` |
| `original_size_hu` | `src/renderer/render_tree.rs::ImageNode` | `None` |

`PageLayerTree` 변환은 `RenderNodeType::Image(ImageNode)`를 `PaintOp::Image`로 그대로 넘긴다. 따라서 값 전달 경로 자체가 아니라 `src/paint/json.rs`의 `PaintOp::Image` JSON serialization 누락이 본질이다.

현재 `PaintOp::Image` JSON은 `base64`, `fillMode`, `originalSize`, `crop`, `effect`, `transform`을 내보내지만 `brightness`, `contrast`는 내보내지 않는다.

## 작업 목표

`PageLayerTree` image paint op JSON만 보고 downstream backend가 core SVG renderer와 같은 이미지 보정 입력값을 재현할 수 있게 한다.

예상 출력 필드:

```json
{
  "type": "image",
  "effect": "grayScale",
  "brightness": -50,
  "contrast": 70,
  "transform": { "rotation": 0.0, "horzFlip": false, "vertFlip": false }
}
```

기본값도 명시적으로 `brightness: 0`, `contrast: 0`을 내보내 downstream 구현이 필드 존재 여부 분기 없이 처리할 수 있게 한다.

## 범위

- Canvas/WebCanvas 이미지 보정 구현 변경
- WASM API surface 추가

## schemaVersion 판단

`PageLayerTree` JSON에 필드가 추가되므로 exported op shape는 바뀐다. 다만 기존 소비자가 알 수 없는 필드를 무시하는 JSON contract라면 additive/backward-compatible 변경으로 볼 수 있다.

초기 판단은 **schemaVersion 유지**다. 이유는 다음과 같다.

- 기존 필드 의미가 바뀌지 않는다.
- 기존 필드가 제거되지 않는다.
- `brightness/contrast` 기본값 `0`은 기존 렌더링 의미와 동일하다.
- downstream strict schema consumer가 있다면 구현 계획서 승인 단계에서 작업지시자 판단으로 bump를 반영한다.

## 위험 영역

- JSON field 추가는 downstream strict schema validator에는 schema mismatch로 보일 수 있다.
- `schemaVersion`을 올리면 기존 downstream consumer가 version guard에 걸릴 수 있다.
- `brightness/contrast`를 기본값일 때 생략하면 downstream 구현 단순화 요구와 어긋난다.
