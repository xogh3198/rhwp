# diff-engine 테스트 시나리오

이 문서는 `diff-engine.ts`의 회귀 테스트/수동 검증에 바로 사용할 수 있는 시나리오 모음입니다.  
구현 코드는 포함하지 않고, **입력 조건 / 변경 내용 / 기대 결과** 중심으로 정리합니다.

---

## 1. 테스트 범위

- 이력관리 경로: `buildSnapshotFromWasm` + `compareSnapshots` (strategy=`identity`)
- 문서비교 경로: `compareDocuments` (strategy=`alignment`)
- 공통 검증 포인트:
  - `DiffItem.kind` (`text`, `table`, `shape`, `chart`, `paragraphMeta`)
  - `DiffItem.severity` (`added`, `removed`, `modified`)
  - 라벨 보정(삭제+추가 병합, 빈 문단 승격, 분할 라벨 교정)
  - 쪽번호/anchor 부가정보(`leftSectionPage`, `rightSectionPage`)

---

## 2. 기본 데이터셋 구성 가이드

- 기준 문서 `BASE`를 하나 만든다.
- `BASE`에서 파생해 변경 버전(`A1`, `A2`...)을 만든다.
- 이력관리 테스트는 가능하면 같은 세션에서 저장한 스냅샷(IR) 기준으로 수행한다.
- 문서비교 테스트는 파일로 내보낸 두 버전(bytes) 기준으로 수행한다.

권장 섹션 구성:

1. 텍스트 중심 섹션(5~8개 문단)
2. 표 섹션(최소 2개 표, 3x3 이상)
3. 도형/이미지/그래프 섹션(각 1개 이상)
4. 반복 문구 섹션(앵커 오탐 방지 검증용)

---

## 3. 이력관리(identity) 테스트 시나리오

## H-01 텍스트 단순 수정

- 변경: 문단 1개에서 단어 1~2개 수정
- 기대:
  - `kind=text`, `severity=modified` 1건
  - removed/added가 발생하지 않음

## H-02 문단 추가

- 변경: 중간 위치에 문단 1개 삽입
- 기대:
  - `kind=text`, `severity=added` 1건
  - 인접 문단이 `moved`로 과검출되지 않음(또는 최소화)

## H-03 문단 삭제

- 변경: 기존 문단 1개 삭제
- 기대:
  - `kind=text`, `severity=removed` 1건
  - 주변 문단의 노이즈 diff가 적어야 함

## H-04 빈 문단 편집

- 변경: 빈 문단에 텍스트 입력(또는 텍스트 삭제로 빈 문단화)
- 기대:
  - 삭제+추가 2건 대신 `modified` 1건으로 표시

## H-05 문단 순서 이동(실제 이동)

- 변경: 같은 텍스트 문단을 아래/위로 크게 이동
- 기대:
  - `kind=paragraphMeta` 이동 감지 항목 생성
  - 단순 reflow와 구분되어 유지

## H-06 삽입으로 인한 reflow(가짜 이동)

- 변경: 상단에 문단 여러 개만 추가(원문 문단 순서는 유지)
- 기대:
  - `moved` 노이즈가 필터링되어 과도하게 나오지 않음

## H-07 표 내부 값 수정

- 변경: 표 셀 1~2개 텍스트 수정
- 기대:
  - `kind=table`, `severity=modified`
  - 가능하면 셀 변경 수/요약이 diff title/preview에 반영

## H-08 도형 속성 수정

- 변경: 도형 크기 또는 회전값 변경
- 기대:
  - `kind=shape`, `severity=modified`
  - removed+added 분리 대신 수정 1건 우선

## H-09 그래프 데이터 수정

- 변경: 그래프 값 일부 변경
- 기대:
  - `kind=chart`, `severity=modified`
  - 타입 유지 시 수정으로 추적

## H-10 이미지 교체

- 변경: 같은 위치 이미지 파일 교체
- 기대:
  - `kind=shape` 또는 매핑된 kind에서 `modified`
  - fallback 매칭 실패 시 added/removed 2건 가능 여부 기록

---

## 4. 문서비교(alignment) 테스트 시나리오

## D-01 의역/부분 수정 문단

- 변경: 의미는 같지만 표현을 바꾼 문단
- 기대:
  - 가능한 `modified`로 매칭
  - `added+removed` 쌍 과다 발생 시 튜닝 후보로 표시

## D-02 중간 삽입 문단

- 변경: 본문 중간에 신규 문단 삽입
- 기대:
  - 삽입 문단은 `added`
  - 기존 문단들이 연쇄적으로 `removed/added`로 깨지지 않음

## D-03 문단 분할

- 변경: 왼쪽 1문단을 오른쪽에서 2문단으로 분할
- 기대:
  - 라벨 순서가 “변경 + 추가”로 자연스럽게 나옴

## D-04 삭제+추가 병합 대상

- 변경: 구조상 같은 슬롯 문단이 수정되었지만 초기 정렬은 분리될 가능성이 있는 케이스
- 기대:
  - 최종 결과에서 `modified-merged` 형태로 병합

## D-05 반복 문구 많은 문서(앵커 스트레스)

- 변경: “1. 개요”, “참고” 등 반복 문구 다수 포함
- 기대:
  - 앵커 오염이 제한되고, 대구간 밀림이 최소화

## D-06 대용량 문서(성능 가드)

- 변경: 문단 수가 큰 문서 비교
- 기대:
  - 타임버짓 내 결과 반환
  - 필요 시 DP→그리디 fallback이 동작하되 결과 품질이 허용 범위

## D-07 표 구조 변경

- 변경: 표 행/열 추가 및 셀 텍스트 일부 수정
- 기대:
  - `kind=table`에서 modified
  - 완전 신규/삭제로만 처리되지 않는지 확인

## D-08 도형 위치 이동 + 속성 변경

- 변경: 동일 도형 위치 이동 및 회전/크기 변경
- 기대:
  - 가능하면 modified 1건
  - fallback 매칭이 적절히 동작

## D-09 그래프 타입 유지 + 데이터 변경

- 변경: 같은 차트 타입에서 시리즈 값만 변경
- 기대:
  - `kind=chart`, `modified`

## D-10 그래프 타입 자체 변경

- 변경: 막대→꺾은선 등 타입 변경
- 기대:
  - `modified` 또는 정책상 added/removed
  - 정책 결과를 문서화해 일관성 유지

---

## 5. 교차 회귀 시나리오 (이력 vs 문서비교 결과 비교)

## X-01 동일 변경, 경로만 다름

- 입력:
  - 케이스 A: same-session 이력 비교(identity)
  - 케이스 B: 파일 업로드 비교(alignment)
- 기대:
  - 핵심 변경 건수(텍스트/표/도형/그래프)가 큰 차이 없이 일관
  - 차이가 있다면 “경로 특성”으로 설명 가능해야 함

## X-02 컨트롤 추적 안정성

- 입력: 표/도형/그래프 혼합 문서에서 소폭 수정
- 기대:
  - 양 경로 모두 removed+added 난발 없이 modified 중심
  - 실패 케이스는 key/fallback 개선 후보로 기록

---

## 6. 체크리스트 템플릿

각 시나리오 실행 시 아래를 기록:

- 시나리오 ID:
- 비교 경로: `identity` / `alignment`
- 입력 문서:
- 기대 diff:
- 실제 diff:
- pass/fail:
- 비고(노이즈, 성능, 재현 조건):

---

## 7. 우선순위(최소 세트)

먼저 자동화/수동 공통으로 최소 아래 8개를 우선 추천:

- H-01, H-02, H-04, H-07
- D-01, D-03, D-04, D-07

이 8개가 통과하면 텍스트/문단/표 + 라벨 보정 핵심 회귀를 대부분 커버할 수 있습니다.
